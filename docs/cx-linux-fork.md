## 1. cx-distro Directory Structure (Confirmed from GitHub)

The cx-distro repo has the following **versioned** directory structure (all confirmed from the GitHub README):

```
cx-distro/
├── .github/workflows/         # CI/CD pipelines
├── apt/                      # APT repo tooling (deb822 format, signed)
│   └── ...                   # Repository signing + management scripts
├── config/                   # Build config JSON files
│   └── ...                   # release-amd64.json, etc.
├── iso/                      # ISO build configuration
│   ├── live-build/          # Debian live-build config (VERSIONED in git!)
│   │   ├── auto/            # Build automation scripts (auto/config, auto/clean, auto/build)
│   │   └── config/          # Package lists, hooks, includes
│   └── preseed/             # Automated installation preseeds (Debian installer)
├── packages/                 # Debian meta-packages
│   ├── cx-archive-keyring/  # GPG keyring for cx APT repo
│   ├── cx-core/             # Minimal installation meta-package
│   └── cx-full/            # Full installation meta-package
├── repository/               # APT repository tooling
│   └── scripts/             # repo-manage.sh
├── sbom/                    # SBOM generation (CycloneDX/SPDX)
├── branding/                 # Plymouth theme, wallpapers
├── scripts/                 # Build automation scripts
│   └── build.sh            # Master build script
├── tests/                   # Verification tests
│   ├── verify-iso.sh
│   ├── verify-packages.sh
│   └── verify-preseed.sh
├── Makefile                 # Build orchestrator — runs lb config + lb build
├── README.md
├── CLAUDE.md
└── LICENSE (BSL 1.1)
```

**CRITICAL:** `iso/live-build/` **IS versioned in git.** This is the actual live-build configuration, NOT generated by `lb config`. The `lb config` command generates binary configuration state, but the source configs are committed.

## 2. The Build System

### 2.1 Makefile Targets

From cx-distro Makefile:

```bash
make help          # Show available targets
make deps          # Install build dependencies
make test          # Run verification tests
make clean         # Clean build artifacts
make iso           # Build the ISO image
make package       # Build .deb packages
```

### 2.2 Build Output

```
output/
├── cx-linux-0.1.0-amd64-offline.iso      # Bootable ISO
├── cx-linux-0.1.0-amd64-offline.iso.sha256 # SHA256 checksum
├── packages/
│   ├── cx-archive-keyring_.deb
│   ├── cx-core_.deb
│   └── cx-full_.deb
└── sbom/
    ├── cx-linux-0.1.0.cdx.json           # CycloneDX SBOM
    └── cx-linux-0.1.0.spdx.json          # SPDX SBOM
```

### 2.3 Distribution

cx-distro uses **Debian 13 "Trixie"** (not Ubuntu). DuckBotOS will change this to **Ubuntu 24.04 LTS "Noble Numbat"**.

Key `lb config` parameters to change:
- `--distribution trixie` → `--distribution noble`
- `--mode debian` → `--mode ubuntu` (live-build supports Ubuntu mode)

## 3. What We Inherit vs. Replace

### Inherit (keep exactly)
- **Makefile build system** — `make deps`, `make iso`, `make package` targets
- **APT repo tooling** — `apt/sign-release.sh`, deb822 format, GPG signing workflow
- **Package skeleton pattern** — `packages/cx-archive-keyring/`, `packages/cx-core/` debian/ structure
- **CI/CD workflows** — `.github/workflows/build-iso.yml`, `installation-tests.yml`
- **SBOM generation** — `syft` + `cyclonedx-cli` in Makefile
- **ISO output layout** — `output/*.iso`, `output/*.sha256`, `output/sbom/`
- **live-build config** — `iso/live-build/auto/`, `iso/live-build/config/` (adapated for Ubuntu)

### Replace wholesale
- **All packages** — `cx-archive-keyring`, `cx-core`, `cx-full` → `duckbotos-*`
- **APT repo URL** — `repo.cxlinux.com` → `repo.duckbotos.com` (or keep configurable)
- **ISO labels + branding** — "CX Linux" → "DuckBotOS"
- **Distribution** — Debian Trixie → Ubuntu Noble
- **Package dependencies** — cx-* packages → duckbotos-* packages

### Replace partially
- **`config/release-amd64.json`** — version, codename, architectures, ISO label
- **CI/CD workflows** — change repo name, artifact upload paths

## 4. Package Structure Pattern

cx-distro packages follow standard Debian meta-package structure:

```
packages/cx-archive-keyring/
├── debian/
│   ├── changelog
│   ├── compat        # debhelper compat level (9)
│   ├── control       # Package metadata + Depends
│   ├── rules         # Build rules (trivial for meta-packages)
│   └── source/
│       └── format   # "3.0 (quilt)"
└── (no source files — pure meta-package)
```

Example `debian/control`:
```
Source: cx-archive-keyring
Section: admin
Priority: optional
Maintainer: CX Linux Team <team@cxlinux.com>

Package: cx-archive-keyring
Architecture: all
Depends: ${misc:Depends}
Description: GPG keyring for CX Linux APT repository
 Installs the GPG public key used to verify packages from the
 CX Linux APT repository.
```

Example `debian/rules` (trivial meta-package):
```makefile
#!/usr/bin/make -f

%:
	dh $@

override_dh_systemd_enable:
	# No systemd services for keyring package
```

Build command: `dpkg-buildpackage -us -uc -b` from within `packages/{pkg}/`

## 6. Three-ISO Build Strategy

Rather than one ISO with complex installer branching, build three separate ISOs:

```
# Hermes-only ISO
duckbotos-hermes-x86_64.iso
  duckbotos-meta-hermes
  duckbotos-keyring
  duckbotos-base
  duckbotos-hermes
  duckbotos-lm-studio
  duckbotos-browseros
  duckbotos-computer-use
  duckbotos-kiosk-hermes

# OpenClaw-only ISO
duckbotos-openclaw-x86_64.iso
  duckbotos-meta-openclaw
  duckbotos-keyring
  duckbotos-base
  duckbotos-openclaw
  duckbotos-lm-studio
  duckbotos-browseros
  duckbotos-computer-use
  duckbotos-kiosk-openclaw

# Both-mode ISO (with GDM session picker)
```

## 7. APT Repository Integration

DuckBotOS packages are built and stored in a local APT repository embedded in the ISO (offline mode):

```
output/
├── duckbotos-hermes-x86_64.iso
└── repo/                          # Embedded APT repo in ISO
    ├── pool/
    │   └── main/
    │       └── d/
    │           └── duckbotos-*/
    ├── dists/
    │   └── noble/
    │       ├── Release
    │       └── main/
    │           ├── binary-amd64/
    │           │   ├── Packages.gz
    │           │   └── Release
    └── db/
        └── packages.db
```

The `apt/` tooling handles signing and repo metadata generation.

### Signed APT Sources

cx-distro uses deb822 format with GPG signing:
```
# /etc/apt/sources.list.d/duckbotos.sources
Types: deb
URIs: https://repo.duckbotos.com
Suites: noble noble-updates noble-security
Components: main
Signed-By: /usr/share/keyrings/duckbotos-archive-keyring.gpg
```

## 8. Build Environment Requirements

Build dependencies (unchanged from cx-distro):

```bash
sudo apt-get install -y \
    live-build \
    debootstrap \
    squashfs-tools \
    xorriso \
    isolinux \
    syslinux-efi \
    grub-pc-bin \
    grub-efi-amd64-bin \
    mtools \
    dosfstools \
    dpkg-dev \
    devscripts \
    debhelper \
    fakeroot \
    gnupg \
    syft \
    cyclonedx-cli \
    python3-pip
```

**Important:** live-build requires a **Linux host** — cannot build on macOS. This is why P2-1 (Linux VM) is a blocker. Building Debian-based ISOs on macOS is not supported by live-build.

**Distribution requirement:** Build on the same distribution as the target (build Ubuntu Noble ISOs on Ubuntu Noble, or use a Docker chroot).

## 9. CI/CD Build Pipeline

cx-distro GitHub Actions workflows to adapt:

- `build-iso.yml` — triggers on push to main, builds ISO, uploads as artifact
- `installation-tests.yml` — runs VMs to test the ISO installs correctly

For DuckBotOS:
- Build all 3 ISO variants
- Run verification tests (ISO boots, services start, browser loads)
- Upload artifacts to GitHub Releases
- Generate SBOM for each ISO

## 10. cx-distro Key Learnings from README

From the cx-distro README, these are confirmed facts:

1. **live-build is versioned** — `iso/live-build/` IS in git, not generated
2. **Distribution is Debian Trixie** — DuckBotOS changes to Ubuntu Noble
3. **ISO mode is hybrid** — `live-image-{arch}.hybrid.iso` supports both USB and DVD
4. **APT repo uses deb822 format** — modern Debian/Ubuntu standard
5. **SBOM uses CycloneDX + SPDX** — both formats generated via `syft`
6. **Preseed is for Debian installer** — DuckBotOS uses Subiquity autoinstall instead
